在应用层开发中，我们往往不关心错误到底是哪种类型，只关心：**“出错了，告诉我为啥，然后退出”**。`anyhow` 完美契合这一点。

### 重构[上一例错误处理](5.2%20问号运算符的错误处理.md)的代码

你需要先在 `Cargo.toml` 中添加 `anyhow = "1.0"`。重构后的代码如下：

```rust
use std::fs;
use anyhow::{Context, Result};
fn get_config() -> Result<i32> {
    let s = fs::read_to_string("config.txt")
        .context("Failed to read config.txt")?;
    let n = s.trim().parse::<i32>()?;
    Ok(n)
}
fn main() {
    match get_config() {
        Ok(n) => println!("数字是: {}", n),
        Err(e) => {
            eprintln!("出错了: {:#}", e);
            std::process::exit(1);
        }
    }
}
```

与上一例相比，重构后的代码精简了许多。甚至可以进一步简化，直接让 main 也返回 Result

```rust
fn main() -> anyhow::Result<()> {
    let n = get_config()?;
    println!("数字是: {}", n);
    Ok(())
}
```

---

### 为什么 `anyhow` 是真实项目中的宠儿？

1. **告别警告**：你不再需要定义 `MyError` 枚举，也就没有了“字段从未被读取”的死代码警告。
2. **错误回溯 (Context)**：
   - 以前报错只显示：`No such file or directory`（谁知道是哪个文件？）。
   - 现在 `anyhow` 配合 `.context()` 会显示「出错了: Failed to read config.txt: No such file or directory (os error 2)」
   - 注意 `eprintln!` 需使用井号占位符，否则不显示 `.context()` 信息 ⚠️
3. **万能适配**：无论底层是 `io::Error`、`ParseIntError` 还是你自己写的错误，`?` 都能一招搞定。它就像一个巨大的“垃圾桶”，能收纳所有实现了 `std::error::Error` 协议的类型。

### 什么时候不该用 `anyhow`？

虽然它很爽，但有一个原则：**“库 (Library) 用 thiserror，应用 (App) 用 anyhow”**。

- 如果你在写一个给别人用的**插件或库**，请不要用 `anyhow`。因为你的调用者可能需要根据具体的错误类型（比如 `MyError::FileMissing`）来决定是重试还是报错。`anyhow` 把错误抹平了，调用者就很难通过代码逻辑进行区分。
