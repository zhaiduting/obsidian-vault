### 1. 它是如何“实例化”的？

与其说它不需要实例化，不如说 **它的名字本身就代表了它的唯一值**。

当你定义 `struct Stopped;` 时，Rust 实际上做了两件事：

1. 定义了一个名为 `Stopped` 的 **类型**。
2. 定义了一个名为 `Stopped` 的 **值（常量）**。

因此，当你写 `let a = Stopped;` 时，你确实是在实例化，只是语法简化到了极致。你不需要写 `Stopped {}` 或 `Stopped()`，直接用名字就行。

```rust
#[derive(PartialEq, Eq, Debug)]
struct Stopped;

fn main() {
    let a = Stopped;
    let b = Stopped{}; // 花括号要不要都可以

    if a == b {
        println!("它们在逻辑上是恒等的！");
    }
    
    println!("{}", std::ptr::eq(&a, &b)); // false
}
```

---

### 2. 所有实例对象都是相同的吗？

**从逻辑和内存的角度看：是的，它们完全相同。**

- **数据层面**：因为单元结构体没有任何字段，所以它不携带任何状态。两个 `Stopped` 实例之间不存在任何差异。
- **内存层面**：这是最酷的地方，单元结构体的大小是 **0 字节**。单元结构体数组的大小也是 0 字节，无论有多少个元素。
- `std::ptr::eq(&a, &b))` 表明：即便区域长度是 0，它们在栈上的“门牌号”序列号通常是不一样的。

---

### 3. 它相当于“单例”吗？

它在行为上非常像单例，但由于 Rust 的所有权机制，它比传统 OOP 的单例更轻量：

- **传统单例**：通常在堆上申请一块内存，所有引用都指向同一个地址。
- **单元结构体**：根本不需要内存地址。你可以拥有无数个 `Stopped` 变量，但编译器在生成机器码时，甚至可能直接把它们优化掉。

**它更像是一个“类型级别的标签”**。就像数学里的符号 $\emptyset$（空集），你在任何地方写的 $\emptyset$ 意义都一样，不需要专门去“申请”一个空集。

---

### 🛠️ 深度对比：单元结构体 vs 单例模式

| **特性**     | **单元结构体 (Rust Unit Struct)** | **传统单例 (Singleton Pattern)** |
| ------------ | --------------------------------- | -------------------------------- |
| **内存占用** | **0 字节**                        | 至少一个指针的大小 + 对象数据    |
| **创建开销** | 零开销（编译器行为）              | 运行时申请内存、初始化           |
| **状态存储** | **无法存储任何数据**              | 可以存储并修改全局状态           |
| **主要用途** | 标记状态、实现 Trait、泛型占位    | 共享全局资源（如数据库连接）     |

---

### 💡 总结

你可以把单元结构体理解为：**一个只有名字、没有身体的幽灵。**

- **它不是单例**，因为你可以随意创建无数个实例（`let x = Stopped; let y = Stopped;`）。
- **但它胜似单例**，因为无论你创建多少个，它们在内存中都不占空间，且逻辑上完全等价。

**它最强悍的用法**，就是我们之前聊到的“状态机”：利用它的“唯一性”在编译阶段卡死逻辑错误，而不产生任何运行时的内存负担。
