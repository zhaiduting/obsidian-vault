既然你现在转变身份为一个工具库（Library）开发者，那么 `thiserror` 就是你的最佳拍档。

与 `anyhow` 那种“把所有错误扔进一个大箩筐”的做法不同，`thiserror` 的核心思想是：**定义清晰、强类型的枚举，让你的用户能精确地知道发生了什么。**

---

### 1. `thiserror` 重构示例

首先，在 `Cargo.toml` 中添加 `thiserror = "1.0"`。

```rust
use std::{fs, io};
use std::num::ParseIntError;
use thiserror::Error;

#[derive(Debug, Error)]
enum MyError {
    #[error("读取配置文件失败: {0}")]
    Io(#[from] io::Error),

    #[error("配置文件内容无效，不是一个数字: {0}")]
    Parse(#[from] ParseIntError),

    // 你也可以定义自己的纯逻辑错误
    #[error("配置值 {0} 超出允许范围 (0-100)")]
    ValidationError(i32),
}

fn get_config() -> Result<i32, MyError> {
    let s = fs::read_to_string("config.txt")?;
    let n: i32 = s.trim().parse()?;
    if n > 100 {
        // 手动返回自定义的验证错误
        return Err(MyError::ValidationError(n));
    }
    Ok(n)
}

fn main() {
    match get_config() {
        Ok(n) => println!("最终数字: {}", n),
        Err(e) => {
            eprintln!("应用错误: {}", e);
            std::process::exit(1);
        }
    }
}
```

---

### 2. 为什么库开发者爱死 `thiserror` 了？

1. **没有死代码警告**：`thiserror` 内部会自动生成读取字段的代码，你会发现之前困扰你的 `field is never read` 警告消失了。
2. **类型安全**：你的用户可以使用 `match` 来针对性地处理错误：

   ```rust
   match get_config() {
       Err(MyError::Io(e)) => { /* 可能是网络驱动器断了，尝试重连 */ }
       _ => { /* 其他错误 */ }
   }
   ```

3. **极简的模板代码**：你不再需要手写几十行 `impl From` 和 `impl Display`，所有的逻辑都通过 `#[error]` 和 `#[from]` 这两个简单的注解搞定。

---

### 3. 如何在 `main` 中使用它？

即使你在底层用了 `thiserror`，在 `main` 函数里你依然可以保持简洁：

```rust
fn main() {
    if let Err(e) = get_config() {
        // 因为 thiserror 实现了 Display，所以 {} 打印出来的是你在 #[error("...")] 里定义的整洁信息
        eprintln!("错误: {}", e);

        // 如果你想看更底层的原始错误（例如具体的 OS Error），
        // 可以使用 std::error::Error 提供的 source() 方法，或者用 anyhow 包装一下
        std::process::exit(1);
    }
}
```

---

### 总结：`anyhow` vs `thiserror`

| **特性**     | **thiserror (工具库专用)**      | **anyhow (应用程序专用)**          |
| ------------ | ------------------------------- | ---------------------------------- |
| **错误类型** | **强类型** (具体的枚举变体)     | **弱类型** (统一的 Error 对象)     |
| **主要目的** | 让调用者能通过 `match` 区分错误 | 让开发者能快速看到错误上下文和堆栈 |
| **代码量**   | 需要定义 `enum`                 | 不需要定义错误类型                 |
| **展示效果** | 结构化，信息受控                | 链式，信息详尽                     |
