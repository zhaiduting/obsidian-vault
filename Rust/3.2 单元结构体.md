**单元结构体 (Unit-like Struct)** 是 Rust 中最简单的结构体形式。它的定义非常“吝啬”，连括号都没有。它不占用任何内存空间（大小为 0 字节），也被称为 **零大小类型 (Zero-Sized Type, ZST)**。

虽然你可以用一个带字段的结构体来代替它，但在以下三个场景中，单元结构体是**逻辑最清晰**、甚至在某些设计模式下是**唯一优雅**的选择：

### 1. 实现“只有行为，没有数据”的特征 (Traits)

这是最常见的情况。如果你需要实现一个 `Trait`（类似接口），但你的实现逻辑完全不依赖任何实例数据，这时就需要单元结构体。

**场景：策略模式 / 插件系统** 假设你在写一个格式化工具，你需要符合某个特征，但你只需要逻辑：

```rust
trait Formatter {
    fn format(&self, input: &str) -> String;
}

// 仅仅为了给这个“转换大写”的算法起个名字
struct UppercaseFormatter;

impl Formatter for UppercaseFormatter {
    fn format(&self, input: &str) -> String {
        input.to_uppercase()
    }
}
```

这里不需要 `name` 或 `age`，你只是需要一个**类型标签**来承载这段逻辑。

### 2. 作为“标记”或“状态” (Marker Types)

在高级编程（如泛型或类型状态模式）中，单元结构体常被用来在**编译期**标记某种状态。

**场景：状态机控制** 如果你想让编译器帮你检查“只有启动了的任务才能停止”，你可以这样做：

```rust
struct Stopped; // 状态 A
struct Running; // 状态 B

struct Task<State> {
    state: State, // 这里 State 只是个标记，不占空间
}

// 只有状态是 Running 的 Task 才有 stop 方法
impl Task<Running> {
    fn stop(self) -> Task<Stopped> {
        Task { state: Stopped }
    }
}
```

在这里，`Stopped` 和 `Running` 不存数据，它们存在的唯一意义就是告诉编译器：“我现在处于这个状态”。

### 3. 定义全局单例或占位符

有时候某些 API 要求你传入一个实现了特定 `Trait` 的对象，但你此时并不关心它的具体内容。

- **测试/Mock：** 在测试中，你需要传入一个对象，但你并不想初始化一大堆复杂的字段，给它一个单元结构体作为“替身”即可。
- **占位：** 在开发初期，你确定需要一个类型，但还没想好要存什么数据。
