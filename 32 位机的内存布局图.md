在标准的 32 位 Linux 操作系统中，一个进程被分配的虚拟地址空间总共是 **4 GB**（从最低的 `0x00000000` 到最高的 `0xFFFFFFFF`）。这 4 GB 被黑白分明地划分为两大核心阵营：**用户空间（3 GB）** 与 **内核空间（1 GB）**。

我们从最高地址（顶端）开始，一层层向下剥离：

### 操作系统内核空间 (OS Kernel Space)

- **地址范围**：高地址的 1 GB 空间（从 `0xC0000000` 到 `0xFFFFFFFF`）。
- **功能职责**：这块区域常驻着操作系统的核心代码、内核数据结构、驱动程序以及每个进程的内核栈。
- **防护机制**：用户态的代码**绝对不能直接读取或写入**这些地址。如果你的程序尝试通过野指针访问这里，硬件（MMU）会瞬间上报异常，导致操作系统抛出 **Segmentation Fault（段错误）** 并强制杀死程序。

### 用户空间 (User Space)

低地址的 3 GB 空间（从 `0x00000000` 到 `0xC0000000`）是属于你编写的程序的舞台。它由以下五个核心段（Segments）纵向排列组成：

#### ① 栈区 (Stack)

- **增长方向**：**向下增长**（向低地址延伸，Grows towards lower memory addresses）。
- **存放内容**：
  - **自动变量 (Automatic variables)**：即函数内部声明的局部变量（局部作用域）。
  - **函数调用上下文**：调用者的返回地址（Caller's return address）、函数参数、寄存器备份等。
- **特点**：由编译器自动管理分配和释放，随着函数的进入和退出自动弹栈/压栈，速度极快，但容量有限（通常系统默认限制为 8MB 左右）。

#### 🛡️ 空白交汇区 (Unallocated Space)

在 `Stack` 和 `Heap` 之间有一块巨大的白色空白区域。栈从上往下长，堆从下往上长。它们两兄弟共享这块生存空间，如果没有互相撞车（即栈溢出 Stack Overflow 或堆溢出 Heap Overflow），程序就能正常运行。

#### ② 堆区 (Heap)

- **增长方向**：**向上增长**（向高地址延伸，Grows towards higher memory addresses）。
- **存放内容**：通过程序员动态申请的内存（在 C 语言中用 `malloc`，C++ 中用 `new`，Rust 中用 `Box` 或 `Vec` 等）。
- **特点**：生命周期完全由程序员控制。申请了就必须手动释放（`free` / `delete`），否则会造成**内存泄漏（Memory Leak）**（注：Rust 则是利用 RAII 机制在作用域结束时由编译器自动插入释放代码）。

#### ③ BSS 段 (Block Started by Symbol)

- **存放内容**：**未初始化**或者**显式初始化为 0** 的全局变量和静态变量（Static variables）。
- **特点**：在可执行文件中它**不占用实际的硬盘空间**，仅仅是一个符号标记。当程序运行时，内核会在真正装载时将这块内存全部**清零（Filled with zeros）**。

#### ④ 数据段 (Data Segment)

- **存放内容**：已经显式初始化（且初始值不为 0）的全局变量和静态变量。
- **特点**：这些变量的值在编译时就已经确定，因此它们必须实打实地被写入二进制文件中，**占用硬盘空间**。程序启动时，这些值会从硬盘原封不动地加载到内存的 Data 段中。

#### ⑤ 代码段 (Text Segment / Code Segment)

- **地址起点**：位于用户空间的最底部（图示约从 `0x08048000` 开始，其下方紧邻一块防止空指针的保留区）。
- **存放内容**：程序的二进制机器镜像（例如 `/bin/ls` 执行文件的 CPU 指令）。
- **特点**：
  - **只读性（Read-Only）**：为了防止程序在运行时意外或恶意地修改自己的指令，这块区域在硬件层面被标记为只读。
  - **可共享性**：如果同一个程序被同时运行了多次（比如开了 5 个终端都在运行 `ls`），它们的 `Text` 段在物理内存中只需要保留一份，多个进程共享即可，极大地节省了内存。

### 总结表

| **内存段 (Segment)** | **管理方式**     | **增长方向**               | **读写权限**      | **是否占二进制文件体积**       |
| -------------------- | ---------------- | -------------------------- | ----------------- | ------------------------------ |
| **Kernel Space**     | 操作系统统治     | N/A                        | 用户态禁止访问    | 属于系统，不属于用户程序       |
| **Stack (栈)**       | 编译器/系统自动  | 向下 (高 $\rightarrow$ 低) | 可读可写          | 否（运行时动态开辟）           |
| **Heap (堆)**        | 程序员手动控制   | 向上 (低 $\rightarrow$ 高) | 可读可写          | 否（运行时动态申请）           |
| **BSS 段**           | 系统在启动时清零 | 固定大小                   | 可读可写          | **否**（只记录所需大小符号）   |
| **Data (数据段)**    | 编译时确定初始值 | 固定大小                   | 可读可写          | **是**（初始值需写进硬盘文件） |
| **Text (代码段)**    | 编译时确定指令   | 固定大小                   | **只读** / 可执行 | **是**（核心的机器指令）       |

### 全局变量存放在哪儿？

未初始化的全局变量存放于 BSS 段；已初始化的全局变量（且初始值**不为零**），存放于数据段（Data Segment）。

在现代内存布局中，为了极致的效率，编译器把全局变量非常精细地拆成了三份。我们用一个 Rust 的例子来对比：

```rust
// 1. 已初始化、且不为 0 $\rightarrow$ 存放在【数据段（.data）】
static mut COUNTER: i32 = 42;

// 2. 未初始化（或显式初始化为 0）$\rightarrow$ 存放在【BSS 段（.bss）】
static mut TOTAL_SCORE: i32 = 0;

// 3. 常量（只读）$\rightarrow$ 存放在【只读数据段（.rodata）】
static GLOBAL_NAME: &str = "Rust";
```

### 为什么要费尽心机把“已初始化”和“未初始化”分开？

既然都是全局变量，凭什么值是 `42` 的放一堆，值是 `0` 的放另一堆？

**答案是为了缩小“二进制可执行文件”的体积。**

#### 如果都在数据段（.data）：

假设你写了一个巨大的全局数组：`static mut BUFFER: [u8; 1024 * 1024] = [1; 1024 * 1024];`（初始化为 1，大小 1MB）。

- 编译器必须把这 1MB 的 `1` 结结实实地写进编译出来的二进制文件里。你的程序还没运行，文件大小就凭空多了 1MB。

#### 如果放在 BSS 段（.bss）：

假设你写的是：`static mut BUFFER: [u8; 1024 * 1024] = [0; 1024 * 1024];`（初始化为 0，或者未初始化）。

- 编译器在生成二进制文件时，**绝对不会**往文件里写 1056720 个 `0`。
- 它只会在可执行文件的控制信息里写上一行：`".bss 区域需要 1MB 空间"`。这个记录只占几个字节！
- 你的可执行文件体积几乎不会变大。

#### 当程序启动时：

操作系统把程序加载到内存中，看到控制信息说 BSS 段需要 1MB，操作系统就会在内存里批一块 1MB 的地方，然后**调用硬件指令（比如 `memset`）瞬间把这块内存全部刷成 0**。
